# Project Documentation

为软件项目创建清晰、完整、用户友好的 README 和系统文档。

## 核心能力

本 skill 帮助你创建包含以下内容的高质量项目文档：

1. **Mermaid 流程图** - 可视化系统运行原理
2. **Step by Step 使用步骤** - 详细的新手引导
3. **配置说明** - 清晰的环境变量和配置文件组织

## 工作流程

### 1. 理解项目

在开始编写文档之前，先理解项目的核心内容：

**必须了解的信息**：
- 项目的核心功能是什么？
- 技术栈是什么？（前端、后端、数据库、外部服务）
- 用户如何使用这个项目？（启动流程、核心操作）
- 需要哪些外部依赖？（API Key、账号、工具）
- 配置项有哪些？（必需 vs 可选）

**信息收集方式**：
- 阅读现有的 README（如果有）
- 查看 package.json 或类似的依赖配置文件
- 查看 .env.example 或配置文件
- 查看主要的代码文件（如 main.ts、app.ts、routes/）
- 询问用户关键信息

### 2. 创建 Mermaid 流程图

根据项目类型选择合适的流程图：

#### 核心流程图（必需）
展示用户从启动到完成核心任务的完整流程。

**何时使用**：所有项目都应该有核心流程图。

**参考模板**：见 [references/mermaid-patterns.md](references/mermaid-patterns.md) 的"核心流程图"部分。

**关键要素**：
- 起点：用户启动系统
- 决策点：登录验证、配置检查等
- 核心操作：主要功能流程
- 终点：任务完成或返回主界面

#### 数据流向图（推荐）
展示系统各组件之间的交互顺序。

**何时使用**：
- 前后端分离的项目
- 涉及多个服务交互的项目
- 需要说明 API 调用流程的项目

**参考模板**：见 [references/mermaid-patterns.md](references/mermaid-patterns.md) 的"数据流向图"部分。

**关键要素**：
- 参与者：用户、前端、后端、数据库、外部服务
- 交互序号：为每个步骤编号
- 请求-响应：清晰标注数据流向

#### 技术架构图（推荐）
展示系统的技术栈和分层结构。

**何时使用**：
- 技术栈复杂的项目
- 需要说明系统架构的项目
- 多层架构的项目

**参考模板**：见 [references/mermaid-patterns.md](references/mermaid-patterns.md) 的"技术架构图"部分。

**关键要素**：
- 分层：前端层、后端层、数据层、外部服务层
- 技术标注：具体的框架和工具
- 依赖关系：箭头表示依赖方向

**完整的 Mermaid 模板和最佳实践**：阅读 [references/mermaid-patterns.md](references/mermaid-patterns.md)

### 3. 编写 Step by Step 使用步骤

按照以下结构组织使用步骤：

#### 前置要求
列出用户需要准备的内容：
- 软件依赖（Node.js、Python 等，包含版本要求）
- 包管理器（npm、pnpm、pip 等）
- 外部账号（如果需要）
- 其他工具

#### 标准步骤序列
1. **克隆项目并安装依赖**
2. **获取外部服务凭证**（如果需要）
3. **配置环境变量**
4. **初始化系统**（如果需要）
5. **启动服务**
6. **使用功能**
7. **验证系统运行**

#### 每个步骤的标准格式
```markdown
### 步骤 N: 动词开头的标题

简短说明这一步的目的（1-2 句话）。

1. 具体操作 1
2. 具体操作 2
3. 具体操作 3

\`\`\`bash
# 如果有命令，提供完整的可复制命令
command --option value
\`\`\`

**预期输出**:
\`\`\`
期望看到的输出内容
\`\`\`

> 💡 **提示**: 额外说明或技巧
> ⚠️ **注意**: 常见错误或注意事项
```

**完整的步骤编写指南**：阅读 [references/step-by-step-guide.md](references/step-by-step-guide.md)

### 4. 组织配置说明

#### .env.example 文件结构

使用清晰的分隔线和分组：

```bash
# ========================================
# 主分类标题
# ========================================
配置项=示例值

# ----------------------------------------
# 子分类标题
# ----------------------------------------
配置项=示例值
```

#### README 中的配置说明

**必需配置项**：
- 明确标注"必需"
- 提供获取方式
- 给出完整示例
- 说明用途

**可选配置项**：
- 明确标注"可选"
- 说明功能影响
- 提供默认值
- 给出配置示例

**多方案配置**：
- 使用表格对比不同方案
- 为每个方案提供完整配置示例
- 说明适用场景
- 标注推荐方案

**完整的配置组织指南**：阅读 [references/config-organization.md](references/config-organization.md)

## 质量检查清单

完成文档编写后，使用以下清单验证质量：

### 流程图检查
- [ ] 核心流程图清晰展示了用户操作流程
- [ ] 数据流向图准确反映了系统交互
- [ ] 技术架构图完整展示了技术栈
- [ ] 所有图表都使用了合适的颜色和样式
- [ ] 图表复杂度适中（不超过 20 个节点）

### 使用步骤检查
- [ ] 前置要求清单完整
- [ ] 每个步骤都有清晰的标题和目的说明
- [ ] 提供了完整的可复制命令
- [ ] 说明了预期输出或结果
- [ ] 包含了常见问题提示
- [ ] 步骤顺序符合实际操作流程

### 配置说明检查
- [ ] 必需配置项和可选配置项分类清晰
- [ ] 每个配置项都有用途说明
- [ ] 提供了获取外部服务凭证的详细步骤
- [ ] 配置文件使用了清晰的分隔线和分组
- [ ] 多方案配置有对比和推荐

### 用户友好性检查
- [ ] 使用了清晰的图标（✅ ⚠️ 💡）
- [ ] 提供了多种方案选择（如果适用）
- [ ] 包含了故障排查提示
- [ ] 语言简洁易懂
- [ ] 所有外部链接都可访问

## 常见场景

### 场景 1: 为新项目创建 README

1. 使用 Glob/Read 工具了解项目结构
2. 查看 package.json、.env.example 等配置文件
3. 创建核心流程图、数据流向图、技术架构图
4. 编写详细的 Step by Step 使用步骤
5. 组织配置说明
6. 使用质量检查清单验证

### 场景 2: 更新现有 README

1. 阅读现有 README
2. 识别缺失或不清晰的部分
3. 补充 Mermaid 流程图（如果没有）
4. 优化使用步骤（添加预期输出、提示等）
5. 重新组织配置说明（使用分隔线、分组）
6. 使用质量检查清单验证

### 场景 3: 优化配置文件

1. 阅读现有 .env.example
2. 按功能分组配置项
3. 添加清晰的分隔线和注释
4. 标注必需 vs 可选
5. 提供获取方式链接
6. 在 README 中添加详细的配置说明

## 最佳实践

1. **用户视角优先** - 从完全不了解项目的新用户角度出发
2. **可操作性** - 所有命令都应该可以直接复制粘贴执行
3. **完整性** - 包含所有必要的信息，不遗漏关键步骤
4. **清晰性** - 使用图表、表格、代码块等提高可读性
5. **验证性** - 提供验证步骤，让用户确认操作成功

## 注意事项

- 不要假设用户有任何背景知识
- 不要跳过任何步骤
- 不要使用模糊的描述（如"配置好环境变量"）
- 不要遗漏预期输出或验证步骤
- 不要忘记添加常见问题提示
